Die Nutzer können die Webseite aufrufen und besuchen. Jeder kann ein neues Spiel öffnen. Es benötigt keinerlei Vorkonfiguration. Alles was eingestellt werden kann ist auf der Seite beschrieben.


Sprints sind immer eine Woche. Am Ende eines Sprints werden die Aufgaben für den nächsten Sprint besprochen. Für eine Story oder Bug wird eine neue Branch aufgemacht. Wenn die Story abgeschlossen ist, wird ein Merge-Request gestellt. Wenn dieser approved wurde, wird die Branch in master gemerged und die Änderungen auf den Server deployed. Storys und größere Bugs sind in Jira hinterlegt. Kleinere Bugs und Verbesserungen sind in GitHub Issues hinterlegt. Dort kann jeder einen Bug oder eine Verbesserungen melden. Des Weiteren kann jeder der möchte, sich dort eine Issue suchen und bearbeiten. Wenn die Issue behoben ist, kann derjenige / diejenige einen Merge-Request stellen. Sobald dieser genehmigt wurde, gehen die Änderungen auf master und somit live.

5. Build / Deploy / Maintanance


Da die Software ein Webanwendung ist, wird direkt die neue Webseite deployed und steht den Kunden / Nutzern sofort zur Verfügung. Dadurch hat jeder Nutzer die akutelle Version mit den neusten Features. Ein manuelles Updaten der Anwendung ist somit nicht nötig.


Die Software ist eine Webseite. Bei einem neuen Release wird die Webseite mit den Neuerungen veröffentlicht und steht sofort allen Nutzern zur Verfügung. Somit haben stets alle Nutzer die neuste Version der Webseite.


Auf einem linuxbasiertem Webserver. Die Anwendung selbst läuft auf einem Tomcatserver mit der Javaversion 11.


Wie wird der Build getriggert?

Der Build wird bei jedem Push auf die jeweilige Feature-Branch bzw. nach einem Pull-Request auf der master-Branch getriggert.

Was wird gebaut( Setup / JAR/Exe / Container)?

Da es sich um eine Webanwendung handelt, wird für das Backend eine WAR (Web Application Archive) erstellt. Diese kann in einem Tomcat-Container deployed werden.

Wo wird gebaut?

Über Github-Actions. Die WAR-File wird dann automatisch auf den Docker-Container deployed.

Wann/Wie oft werden welche Tests ausgeführt?

Bei jeder Änderung wird entweder die Feature-Pipeline oder die Master-Pipeline angestossen. Beide führen zuerst den Build-Step durch. Wenn dieser erfolgreich abgeschlossen ist, werden alle vorhanden Test im Test-Step durchlaufen. Wenn der Build fehlgeschlägt, stoppt die Pipeline.

Wer wird wann informiert Entwickler/Kunde/DevOps

Kunden werden momentan nicht informiert, da diese im Normalfall von den Änderungen nichts mitbekommen. Zukünftig sollen die Nutzer allerdings bei einem neuen Relaise der Seite über neue Features informiert werden. Entwickler bekommen eine Benachrichtung, falls ihre angestossene Pipeline fehlschlägt, da sich diese manuell darum kümmern müssen.

Script für Semantic Versioning erstellen
Testautomatisierung in der Builpipeline

Tests werden automatisch bei jeder neuen gepushten Änderung mit erfolreichem Build durchlaufen. Dadurch wird sichergestellt, dass die Änderungen immer noch alle Tests erfüllen.

Wer kümmert sich um was? Vorgehen festlegen, z.B.:
Agiles vorgehen: welche Sprint Zyklen

Wir haben uns auf einen Sprint-Zyklus von einer Woche geeinigt. Im Laufe der Zeit wurden die Ideen und Feauters im Backlog gesammelt. Am Ende eines Sprints wird besprochen, welche Aufgaben in den neuen Sprint kommen. Dadurch ist eine flexible Arbeitsweise und eine schnelle Anpassung auf den aktuellen Stand möglich. Bevor eine Aufgabe in den Sprint kommt, wurde diese von uns beiden geschätz. Dadurch konnte abgewät werden, welche und wieviele Aufgaben wir in den Sprint mit aufnehmen können. Kanban/Scrum Board aufsetzen:

Für das Projekt wird ein Scrum-Board in Jira verwendet. In diesem werden die Storys und technischen Aufgaben gesammelt und geschätz.Für Bugs und Verbesserungsvorschläge von Nutzern können über GitHub-Issues eingegreicht werden. Dadurch ist es für den Nutzer, der die Issue einreicht, transparent gehalten.
